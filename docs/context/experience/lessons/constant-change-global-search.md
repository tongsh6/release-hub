# 修改常量/错误码时必须全局搜索

## 问题

将错误码从 `GROUP_008` 改为 `GROUP_014` 后，没有搜索所有引用位置，导致 3 个测试文件中的断言仍在使用旧错误码 `GROUP_008`，测试失败。

## 规则

**修改任何常量、错误码、枚举值时，必须执行以下步骤：**

1. **全局搜索旧值** — 使用 Grep 搜索所有引用（包括测试代码、配置文件、注释）
2. **逐一确认** — 判断每个引用是否需要同步更新
3. **一次性全部更新** — 在同一次修改中更新所有引用
4. **运行测试验证** — 确保没有遗漏

### 搜索范围必须包括

- `src/main/java` — 业务代码
- `src/test/java` — 测试代码（最容易遗漏）
- `src/main/resources` — 配置文件、i18n 消息
- 前端代码 — 如果前后端共享常量
- 文档 — API 文档、README 等

## 类似场景

- 重命名方法/类/字段
- 修改 API 路径
- 修改数据库表名/列名
- 修改 i18n message key
- 修改枚举值

## 工具支持

```bash
# 搜索错误码引用
Grep pattern="GROUP_008"

# 搜索枚举值引用
Grep pattern="ErrorCode\.GROUP_HAS_CHILDREN"
```

## 额外教训：错误码语义匹配

不要复用语义不匹配的错误码：
- `GROUP_008` 消息是 "Cannot delete group with children" — 删除场景
- 但被用于 "不能在非叶子节点创建资源" — 创建场景

**应该为不同业务语义创建独立的错误码**，即使底层检查逻辑相似。
