# ReleaseHub 用户故事

> 版本：v1.3  
> 最后更新：2026-01-28  

## 概述

本文档定义了 ReleaseHub 核心功能的用户故事，作为功能实现和自动化测试的依据。

## 角色定义

| 角色 | 描述 |
|---|---|
| **项目管理人员** | 规划发布窗口与迭代，维护上线时间与窗口状态，调整挂载迭代与发布范围 |
| **技术负责人** | 决定迭代/修复类型并挂载仓库，触发提测合并与上线收尾编排，保障发布闭环 |
| **测试人员** | 跟踪上线时间、发布迭代与关联仓库，查看发布计划与执行进度 |
| **系统管理员（运维）** | 维护分组、仓库、分支规则与 GitLab 连接配置，确保基础配置可用 |

## 通用约定

- **分组选择**：发布窗口/迭代/仓库创建与修改时需选择分组末端节点，提交 `groupCode`
- **分支命名**：`feature/<iterationKey>`、`hotfix/<iterationKey>`、`release/<windowKey>`
- **分支归档**：重命名为 `archive/<reason>/<original>`，`<reason>` ∈ {`unpublished`, `released`}，`<original>` 不含空格且遵循分支命名

## US-RW: 发布窗口管理

覆盖发布窗口的创建、状态流转与迭代挂载调整。

### US-RW-001: 创建发布窗口

**作为**项目管理人员  
**我希望**创建一个新的发布窗口  
**以便**规划上线时间并管理即将进行的发布

**验收标准：**
- [x] 可以输入窗口名称
- [x] 创建后状态为待发布
- [x] 系统自动生成唯一 windowKey
- [x] 可填写发布窗口的简短描述
- [x] 可以设置发布窗口的上线时间
- [x] 创建与修改时需选择分组末端节点
- [x] 可先创建发布窗口，后续再挂载迭代

**API：**
```
POST /api/v1/release-windows
PUT  /api/v1/release-windows/{id}/window
```

---

### US-RW-002: 发布窗口状态流转

**作为**项目管理人员  
**我希望**管理发布窗口的生命周期  
**以便**控制发布流程的各个阶段

**状态流转：**
```
待发布 → 已发布 → 已关闭
```

**验收标准：**
- [x] 创建后状态为待发布
- [x] 待发布 → 已发布: 发布窗口
- [x] 已发布窗口可执行版本更新
- [x] 已发布 → 已关闭: 上线完成并完成收尾动作后关闭窗口
- [x] 状态变更有时间记录

**API：**
```
POST /api/v1/release-windows/{id}/publish
POST /api/v1/release-windows/{id}/close
```

---

### US-RW-003: 冻结/解冻发布窗口

**作为**项目管理人员  
**我希望**临时冻结发布窗口  
**以便**在紧急情况下暂停发布活动

**验收标准：**
- [x] 冻结后禁止新的版本更新
- [x] 冻结后禁止配置变更
- [x] 冻结后禁止调整挂载迭代
- [x] 可以解冻恢复正常操作
- [x] 记录冻结/解冻时间

**API：**
```
POST /api/v1/release-windows/{id}/freeze
POST /api/v1/release-windows/{id}/unfreeze
```

---

### US-RW-004: 调整发布窗口挂载迭代

**作为**项目管理人员  
**我希望**在发布窗口中调整挂载的迭代  
**以便**根据实际进度动态调整发布范围

**验收标准：**
- [x] 支持为同一发布窗口新增迭代挂载
- [x] 支持从发布窗口移除已挂载迭代
- [x] 调整后发布计划可实时更新
- [x] 迭代挂载到发布窗口时自动创建 release 分支
- [x] 解除挂载时自动归档对应 release 分支，归档原因为 `unpublished`
- [x] 归档规则遵循通用约定

**API：**
```
POST /api/v1/release-windows/{id}/attach
POST /api/v1/release-windows/{id}/detach
GET  /api/v1/release-windows/{id}/plan
```

## US-IT: 迭代管理

用于规划迭代周期与仓库挂载，是发布窗口范围的基础。

### US-IT-001: 创建迭代

**作为**项目管理人员  
**我希望**创建迭代  
**以便**组织一个冲刺周期内的代码变更

**验收标准：**
- [x] 系统自动生成唯一 iterationKey
- [x] 可填写名称、描述与预期上线时间
- [x] 预期上线时间仅用于参考提示，不做逻辑限制
- [x] 创建时可不关联仓库
- [x] 创建与修改时需选择分组末端节点
- [x] 迭代可关联到发布窗口

**API：**
```
POST /api/v1/iterations
PUT  /api/v1/iterations/{key}
```

---

### US-IT-002: 关联迭代到发布窗口

**作为**项目管理人员  
**我希望**将迭代首次关联到发布窗口  
**以便**确定初始发布范围

**验收标准：**
- [x] 可以关联多个迭代到一个窗口
- [x] 首次关联要求发布窗口处于“待发布”状态
- [x] 关联后自动包含迭代的所有仓库
- [x] 关联时提示迭代预期上线时间与发布窗口上线时间的差异（不强制拦截）
- [x] 可以查看发布计划（Plan）
- [x] 后续调整挂载见 US-RW-004

**API：**
```
POST /api/v1/release-windows/{id}/attach
GET  /api/v1/release-windows/{id}/plan
```

---

### US-IT-003: 挂载仓库到迭代

**作为**技术负责人  
**我希望**为迭代挂载相应的代码仓库  
**以便**明确本次迭代的发布范围与代码变更

**验收标准：**
- [x] 支持为同一迭代添加多个仓库
- [x] 支持从迭代中移除仓库
- [x] 迭代详情可查看已挂载仓库列表
- [x] 允许同一仓库挂载到多个迭代
- [x] 挂载/解除挂载不做迭代状态限制
- [x] 挂载仓库时可选择“迭代”或“修复”类型
- [x] 选择“迭代”时从 master 创建 `feature/<iterationKey>` 分支
- [x] 选择“修复”时从 master 创建 `hotfix/<iterationKey>` 分支
- [x] 解除挂载时自动归档对应 feature/hotfix 分支，归档原因为 `unpublished`
- [x] 归档规则遵循通用约定

**API：**
```
POST /api/v1/iterations/{key}/repos/add
POST /api/v1/iterations/{key}/repos/remove
GET  /api/v1/iterations/{key}
```

---

### US-IT-004: 删除迭代

**作为**项目管理人员  
**我希望**删除不再需要的迭代  
**以便**保持迭代列表简洁

**验收标准：**
- [x] 未关联发布窗口且未挂载仓库的迭代可删除
- [x] 已关联发布窗口或已挂载仓库的迭代不允许删除

**API：**
```
DELETE /api/v1/iterations/{key}
```

## US-REPO: 代码仓库管理

**作为**系统管理员（运维）  
**我希望**添加 GitLab 代码仓库  
**以便**将仓库纳入发布管理

**验收标准：**
- [x] 仓库名称必填

- [x] 默认分支可不填，未填写时自动解析 main 或 master 作为默认分支

- [x] 创建与修改时需选择分组末端节点

**API：**
```
POST /api/v1/repositories
```

## US-BR: 分支规则管理

约束分支命名规则，并在关键节点触发校验。

### US-BR-001: 管理分支规则

**作为**系统管理员（运维）  
**我希望**创建和维护分支规则  
**以便**统一分支命名规范并降低违规风险

**验收标准：**
- [x] 可以创建、编辑、删除分支规则
- [x] 当前仅固定分支格式：`feature/<iterationKey>`、`hotfix/<iterationKey>` 与 `release/<windowKey>`

**API：**
```
POST /api/v1/branch-rules

GET  /api/v1/branch-rules
GET  /api/v1/branch-rules/paged
```

- [x] 创建分支时强制校验命名规则
- [x] 挂载迭代/发布窗口时校验分支命名规则

## US-REL: 发布准备与提测

用于生成测试版并完成提测准备，发生在上线前。

### US-REL-001: 合并迭代代码到 release

**作为**技术负责人  
**我希望**在需要提测或发布测试版本时将迭代代码合并到 release 分支  
**以便**生成可部署的测试版本

**验收标准：**
- [x] 迭代挂载到发布窗口时，若已存在对应的 `feature/<iterationKey>`，创建 release 分支后自动合并
- [x] 技术负责人可手动触发合并指定迭代
- [x] 支持一键合并窗口内全部迭代
- [x] 返回合并结果列表
- [x] 合并目标分支为 `release/<windowKey>`

**API：**
```
POST /api/v1/release-windows/{windowId}/iterations/{iterationKey}/merge
POST /api/v1/release-windows/{windowId}/merge
```

## US-POST: 上线后收尾

上线完成后、关闭窗口前的一键收尾与归档流程。

### US-POST-001: 发布收尾编排与任务追踪

**作为**技术负责人  
**我希望**在发布窗口上线完成、关闭窗口前执行收尾编排并追踪任务状态  
**以便**完成迭代收尾与发布闭环

**验收标准：**
- [x] 一键触发收尾编排任务
- [x] 后台异步执行并有进度提示
- [x] 可在运行记录中查看进度状态与结果
- [x] 运行记录展示总体状态（RUNNING/SUCCESS/FAILED/PARTIAL）
- [x] 运行任务列表展示每个步骤状态与失败原因
- [x] 运行任务列表展示迭代归档步骤与归档结果
- [x] 任务失败可重试
- [x] 收尾编排包含 release 合并到 master
- [x] 合并完成后自动归档 feature/hotfix 分支，归档原因为 `released`
- [x] 收尾编排包含迭代归档
- [x] 归档规则遵循通用约定

**API：**
```
POST /api/v1/release-windows/{id}/orchestrate
GET  /api/v1/runs/{id}
GET  /api/v1/runs/{runId}/tasks
POST /api/v1/runs/{id}/retry
```

说明：迭代归档由收尾编排任务执行，无独立 API。
