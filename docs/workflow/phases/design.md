# Design Phase

## 触发条件

- Proposal 阶段已完成，且满足任一条件：
  - 跨模块变更（涉及 2+ 后端模块或前后端联动）
  - 引入新外部依赖
  - 数据库迁移复杂（非简单 ALTER）
  - 安全敏感（认证/授权/加密变更）
  - 性能敏感（批量操作、高频接口）

## 跳过条件

- 单模块内的简单功能添加
- 接口格式变更（不涉及业务逻辑）
- Proposal 中已包含足够的技术细节

## 执行 Agent

→ [`.ai/agents/design.md`](../../.ai/agents/design.md)

## 执行步骤

1. **加载上下文**：阅读 proposal.md、相关领域模型、架构文档
2. **加载经验**：检索 `context/experience/INDEX.md` 中的相关经验
3. **编写设计文档**：补齐 `openspec/changes/<change-id>/design.md`
   - 技术决策与备选方案
   - 模块间依赖与接口契约
   - 数据库迁移计划（如涉及）
   - 风险与回滚策略
4. **校准完整蓝图**：确认最终架构、完整能力范围、阶段边界、未完成项追踪位置没有缺口
5. **绘制任务 DAG**：明确各垂直切片之间的依赖、可并行点和必须串行的原因
6. **更新任务清单**：根据设计细化 `tasks.md`，每个 Slice 必须可独立验收，并能追溯到完整蓝图

## 产出物

| 文件 | 必须 | 说明 |
|------|------|------|
| `design.md` | 是 | 技术设计文档 |
| `tasks.md`（更新） | 是 | 细化后的任务清单 |

## 验证标准

- `design.md` 包含明确的技术决策（不只是描述，要有结论）
- 备选方案至少列出 2 个（含被否决的方案及否决理由）
- `design.md` 包含完整目标蓝图与最终架构，不得只覆盖本次会话能完成的部分
- 迁移步骤（如涉及）可逐步执行和回滚
- `tasks.md` 的任务颗粒度可分步推进；单次会话未完成的任务必须保留状态和后续顺序
- 任务 DAG 无循环依赖，且每条边都有业务或技术原因
- 每个 Slice 通过事前约束检查：蓝图归属、用户价值、端到端路径、单一目标、可验证、可回滚、依赖明确、风险收敛、后续项清晰
