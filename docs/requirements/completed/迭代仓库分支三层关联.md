# 需求：迭代-仓库-分支 三层关联与 feature 分支创建模式

> **状态：✅ 已完成（2026-05-11）**
> 实现 commit：`931a6b3`（功能落地）+ `0e2efb3`（多 Provider 端口收口）
> 真实 GitLab 验收：`docs/reports/acceptance-v0.1.11-real-gitlab.md` 场景 10.1-10.5 全 PASS
> OpenSpec 提案：`docs/openspec/changes/add-branch-creation-mode/`

## 背景

当前迭代与仓库的关联只有一层：`Iteration.repos: Set<RepoId>`，只能表达"这个迭代包含哪些仓库"，不能表达"每个仓库使用哪个 feature 分支以及以什么方式创建"。feature 分支名通过 `IterationRepoVersionInfo.featureBranch` 间接存储，且分支创建逻辑在 `addRepos()` 中硬编码为 `feature/{iterationKey}`，`create()` 路径完全没有分支映射和分支创建能力。

这导致：
- 编排时找不到 feature 分支（因为默认规则 `feature/{iterationKey}` 与开发者实际的分支名不匹配）
- 开发者无法自定义 feature 分支名
- 开发者已创建的分支无法被系统识别和关联
- `create()` 和 `addRepos()` 行为不对称，`create()` 漏掉了分支映射

## 目标

将迭代-仓库关联升级为三层模型（迭代 → 仓库 → 分支 → 创建方式），提供三种 feature 分支创建模式（AUTO / NAMED / EXISTING），统一 `create`/`addRepos`/`update` 三个入口的行为。

## 三层数据模型

```
Iteration
  └── repoAssociations: List<RepoAssociation>    // 第一层：仓库
        └── repoId: String                        // 仓库 ID
        └── featureBranch: String                 // 第二层：分支名
        └── branchCreationMode: enum              // 第三层：分支创建方式
```

### 分支创建模式 (BranchCreationMode)

| 模式 | 含义 | 系统行为 |
|------|------|---------|
| `AUTO` | 系统自动创建 | `createBranch("feature/{iterationKey}")` + 写入 `featureBranch` |
| `NAMED` | 用户自定义名称 | `createBranch(用户输入的名称)` + 校验在 `feature/` 下 + 写入 `featureBranch` |
| `EXISTING` | 关联已有分支 | 不创建，校验分支存在于 GitLab + 校验在 `feature/` 下 + 写入 `featureBranch` |

### 系统兜底策略

所有分支关联操作必须通过以下校验：
- 分支名必须以 `feature/` 开头（`master`/`main`/`develop` 天然被此规则排除）
- 分支名不包含 `release/`、`hotfix/` 保护前缀（防御性编程）
- 分支名符合 BranchRule 的 TEMPLATE/REGEX 规则
- `EXISTING` 模式下，分支必须确实存在于 GitLab

## 验收标准

- [ ] 创建迭代时可以为每个仓库选择分支创建模式（AUTO/NAMED/EXISTING）
- [ ] AUTO 模式自动创建 `feature/{iterationKey}` 分支
- [ ] NAMED 模式校验用户输入必须以 `feature/` 开头，调用 createBranch
- [ ] EXISTING 模式列出仓库的 `feature/*` 分支供选择，不创建新分支
- [ ] 所有模式写入 `IterationRepoVersionInfo.featureBranch`
- [ ] `feature/` 路径外的分支不可选、不可输入
- [ ] 新增 API：列出仓库 `feature/` 路径下的分支列表
- [ ] `create()`、`addRepos()`、`update()` 三个入口行为一致
- [ ] 移除仓库时归档对应的 feature 分支
- [ ] Attach 前校验 `featureBranch` 不为空
- [ ] 后端 Surefire 测试通过
- [ ] 前端分支选择 UI 可用

## 排除范围

- 不改变 release 分支的创建逻辑（Attach 时创建，编排时复用）
- 不改变 tag、CI、归档等收尾流程
- 不在此需求中引入 feature 分支的 MR 创建或 Code Review 集成

## 技术方案

涉及模块：
- `IterationAppService`：重构 `create`/`addRepos`/`update`，统一调用 `setupRepoForIteration`
- `IterationRepoPort` / `IterationRepoVersionInfo`：扩展支持 `BranchCreationMode`
- 新增 API：`GET /api/v1/repositories/{id}/branches?prefix=feature/`
- 前端：迭代创建/编辑页面新增分支创建模式选择组件

方案细节见 `openspec/changes/add-branch-creation-mode/proposal.md`
